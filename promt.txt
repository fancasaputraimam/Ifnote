PART 3 — KOTOBA LOOKUP BEFORE AI
==================================================

When user uses Tambah Kotoba → AI Analyze:

First call saved-data lookup.

Preferred backend endpoint:

GET /api/kotoba/lookup?q=...

Backend behavior:
- use authenticated userId
- search user's saved Kotoba first
- match by:
  - exact jp
  - normalized jp
  - reading if field exists
  - meaning
  - normalized meaning
  - romaji if available
  - optional contains/fuzzy search

If found, return:

{
  "found": true,
  "item": { ...kotoba }
}

If not found:

{
  "found": false
}

Frontend behavior:
- If found:
  - show preview from saved Catatan
  - badge: "Ditemukan di Catatan"
  - helper: "Catatan ini sudah pernah kamu simpan."
  - do NOT call AI
  - buttons:
    - Buka Catatan
    - Gunakan Data Ini
    - Batal

- If not found:
  - call AI analyze endpoint
  - show AI preview
  - badge: "Hasil Analisa AI"
  - helper: "Periksa hasilnya sebelum disimpan."
  - buttons:
    - Simpan
    - Edit
    - Batal

Acceptance:
- Input "食べます" if exists → saved result appears.
- Input "makan" if meaning exists → saved result appears.
- Input new word → AI analysis appears.
- UI must not say "AI tidak dipakai" or "database".

PART 5 — BULK KOTOBA DATABASE-FIRST
==================================================

Bulk Kotoba AI Analyze must check saved data before AI for each item.

Flow:
1. User enters up to 50 items.
2. Normalize each input.
3. Backend receives items.
4. Backend checks saved Kotoba first for each item.
5. Items found in saved Catatan:
   - status = "exists"
   - return existing item
   - do NOT send those items to AI internally
6. Items not found:
   - send only missing items to AI
7. AI analyzes missing items.
8. Validate AI results.
9. Return combined preview.

Important:
- Do not call AI for items that already exist.
- Only call AI for missing items.
- Check duplicates inside the same bulk request too.
- Max 50 still enforced.

Response item shape:

{
  "sourceInput": "berat",
  "status": "new" | "exists" | "manual" | "duplicate",
  "jp": "重い",
  "reading": "おもい",
  "meaning": "berat",
  "type": "kata sifat い",
  "level": "N5",
  "existingId": null
}

For existing:

{
  "sourceInput": "食べます",
  "status": "exists",
  "existingId": "xxx",
  "jp": "食べます",

PART 6 — KANJI DATABASE-FIRST / CACHE-FIRST
==================================================

When user clicks a kanji:

Frontend:
- open KanjiPopup immediately
- show loading
- call backend:

GET /api/kanji/:kanji

Backend behavior:
1. Normalize kanji.
2. Check kanji_cache for current user.
3. If found:
   - return cached detail
   - do NOT call AI internally
4. If not found:
   - call AI or fallback dictionary
   - validate result
   - save to kanji_cache
   - return result

Response:

{
  "kanji": "食",
  "meaning": "makan",
  "onyomi": "しょく",
  "kunyomi": "たべる",
  "explanation": "Kanji yang berhubungan dengan makan.",
  "words": [
    { "jp": "食べます", "reading": "たべます", "meaning": "makan" }
  ],
  "exampleJp": "ごはんを食べます。",
  "exampleReading": "ごはんをたべます。",
  "exampleMeaning": "Saya makan nasi."
}

Frontend:
- show normal Kanji detail only
- do NOT show technical source/cache/database text
- buttons/labels:
  - Arti
  - Onyomi
  - Kunyomi
  - Penjelasan
  - Contoh kata
  - Contoh kalimat

Acceptance:
- Click same kanji twice.
- First time may call AI/fallback and save.
- Second time uses saved cache internally.
- User only sees normal kanji detail.
